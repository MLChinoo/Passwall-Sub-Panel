# PSP v3.9.0 设计：客户端模型与 3X-UI 同构（一个 client 挂多 inbound）

> 接续 [3xui-3.2-clients-migration.md §4.2 的「v4 方向」备注](3xui-3.2-clients-migration.md#L128)。
> 目标版本 **v3.9.0**。核心原则：**让 PSP 的客户端存储模型与 3X-UI 3.x 同构，从根上消除"不对称"**
> —— 不对称正是 drift、流量双重计数、reconcile 翻译层复杂度的病根。
>
> 本设计的两个关键决策已拍板：
> - **凭据：方案 A（全对称）** —— PSP 存实际凭据、render 直接读，退役按 UUID 现算的 `DeriveProxyPassword`。
> - **版本：v3.9.0**，PSP 面板版本号随之。

---

## 0. 真机已验证的前提（3.3.1，xray 26.6.1，2026-06）

在 `198-23-159-183` 面板上用 API token 实测，结论作为本设计的事实基础：

1. **一个 client 挂多 inbound 是一等公民**：`POST /clients/add {client, inboundIds:[1,2]}` 一次建成；
   `attach`/`detach` 增量挂载/摘除；`add/update/attach/detach` 自 **3.1.0** 即有（见
   [3xui-3.2-clients-migration.md:40](3xui-3.2-clients-migration.md#L40)），PSP floor 3.2.0 全程覆盖 →
   **不用抬 `version.MinXUI`，不用双路兼容**。
2. **投影模型**：一条 client 主记录被投影进它所挂每个 inbound 的 `settings.clients[]`，按协议各取所需字段。
   实测同一 client（同 `id`/UUID）在 VLESS inbound 只出现 `id`，在 SS-2022 inbound 额外出现 `password`。
3. **流量按 email 聚合**：注入一次 111M/222M，**两个 inbound 的 clientStats 都显示同一个 111M/222M**。
   `/clients/list` 每 client 只有**一份** `traffic{up,down}` + `inboundIds[]`。
   → **现状 PSP「按 inbound 求和 owned client」会双重计数**（[traffic.go:492](../internal/service/traffic/traffic.go#L492)）。
4. **inbound 自身 `up/down` 计数器在 3.3.1 可靠**（两个 inbound 都有真实值）→ 节点级流量可改读它。
5. **服务端可生成凭据**：`add` 省略 `uuid/password/auth` 时面板自动生成并存到主记录，投影到所有挂载点。

## 1. 现状（要被替换的）

- **ownership 表 `user_xui_clients`（`domain.XUIClientEntry`）**：唯一键
  `(PanelID, InboundID, ClientEmail)`，**一行 = 一个 (user, node)**。存 `ClientUUID`(= user.UUID) +
  生命周期/周期 流量计数。**不存 password**（[types.go:600](../internal/domain/types.go#L600)）。
- **email = `u{uid}-n{nodeID}@domain`**（[types.go:238](../internal/domain/types.go#L238)），nodeID 嵌入是为兼容
  「全局唯一 email」的 fork。
- **凭据现算**：render 三条路径（[protocols.go](../internal/service/render/protocols.go)、`singbox.go`、`urilist.go`）
  全部 `crypto.DeriveProxyPassword(u.UUID, protocol, method)`；VLESS/VMess/Hy2 直接用 `u.UUID`。
  sync 推送时也用同一函数 → 「同一函数保两端一致」的 lockstep，是**第二个真相源**。
- **3X-UI 侧**：每 (user, node) 一个独立 client（email 不同）。一个面板 N 个节点 → 同一用户 N 个 client。

## 2. 目标模型（与 3X-UI 同构）

| 3X-UI 3.x | PSP v3.9.0（新） |
|---|---|
| `clients` 主表（唯一 email，存 uuid/password/auth + 配额/到期） | **`psp_client`**：一行 = `(userID, panelID[, credClass])`，存 email/uuid/password/auth + 流量计数 |
| `client_inbounds` 中间表 `(clientId, inboundId, flowOverride)` | **`psp_client_inbound`**：`(clientID, nodeID, flowOverride)` |
| 每 email 一份流量 | `psp_client` 上一份流量计数 |

- **email = `u{uid}@domain`**（去掉 nodeID）。面板内按用户全局唯一，正好契合 first-class client 的
  email 唯一约束；「全局唯一 email」的 fork 反而更自然。
- **挂载集 = 该用户在该面板上可访问的节点**（由 group `TagFilter` 决定）。`psp_client_inbound` 存**期望挂载集**，
  reconcile 与 3X-UI `GetClient().inboundIds` 做 attach/detach 差分。

### 2.1 `credClass` —— 唯一不可约的拆分

一个 3X-UI client 只有**一个** `password` 字段。方案 A 下 PSP 存一个 password 值 `P`：

- 令 `P` = **SS-2022-256 PSK 格式**（base64 of 32 bytes，即今天 SS-2022 的派生值）。
- `P` 同时是合法的 **Trojan / 普通 SS / SS-2022-256** 凭据（Trojan/SS password 自由形）。
- VLESS/VMess 用 `id`(=UUID)，Hy2 用 `auth`(=UUID)，都不碰 `password`。

→ **`{VLESS, VMess, Trojan, SS, SS-2022-256, Hy2}` 全部共享一个 client，零冲突。**
**唯一**需要单列第二个 client 的情形：同一用户在同一面板上**同时**有 **SS-2022-128（16B PSK）和 SS-2022-256（32B PSK）** 的 inbound
（两种 PSK 字节长不同，挤不进一个 password 字段）。极罕见。

因此 `credClass` 取值通常恒为 `0`（默认类），仅当出现上述 128/256 混用时分出 `1`。
绝大多数用户、绝大多数面板 = **一个用户一个面板一个 client**。

## 3. 凭据全对称（方案 A）

- `psp_client` 存 `UUID`、`Password`、`Auth`（镜像 3X-UI 主记录）。
- **生成（一次性、确定性、但结果落库）**：`UUID` 仍是 v4；`Password = base64(sha256(UUID))[:32]`（沿用今天
  SS-2022 的派生，但**只算一次、存下来、所有 password 协议统一用**）；`Auth = UUID`。
  确定性让迁移可离线计算、无需回读面板。
- **render 改为读存值**：三条渲染路径不再调 `DeriveProxyPassword`，改读 `psp_client.{UUID,Password,Auth}`。
  - VLESS/VMess：`id = UUID`；Hy2：`auth = UUID`。
  - Trojan/SS：`password = Password`。
  - SS-2022：`password = <inbound serverPSK>:<Password>`（与今天 `singbox.go:194` 同构，user PSK 改读存值）。
- **`crypto.DeriveProxyPassword` 退役**（保留一个一次性生成器 `NewProxyPassword(uuid)` 给建号/迁移用）。
  这样**只剩一个真相源**（`psp_client`），lockstep 隐患消失，SS-2022 不再是 render 特例。
- **`flowOverride`**：VLESS flow 差异（vision vs 空）落到 `psp_client_inbound.flowOverride`，render 按挂载点取。

## 4. 流量模型（两处都改，否则双重计数）

- **用户级**：按 email **读一次 client**（`/clients/get` 或 `/clients/list` 的单份 `traffic`），写到 `psp_client`
  的生命周期/周期计数。**删除**「按 inbound 求和 owned client」逻辑。
- **节点级**：改读 **inbound 自身 `up/down`**（`ports.Inbound.Up/Down`，`ListInbounds` 已返回）。
  语义微调：含该 inbound 上**非 PSP 托管**的 client（PSP 独占 inbound 的部署无差别）。
- **迁移**：把同一 (user, panel) 的 N 行 per-node 计数**合并**成一行（求和生命周期值、取 max LastRaw 作新基线）。
  存在一次**计数断点**，记录在 UPGRADE note。

### 4.1 「每用户 × 每 Server」用量 —— 模型天然支持（一等能力）

我们**保留并明确**这条粒度：**看到每个用户在每个 Server（面板）上的用量**。这正是 v3.9.0 模型的天然单位——
一个 `psp_client` = 一个 `(user, panel)`，而 3X-UI 对一个 email 只报**一份**聚合流量（实测每个挂载 inbound
回显同一个数），所以 **`psp_client` 的生命周期/周期计数本身就是「该用户在该 Server 的用量」**，零额外成本。

- **当前用量**：`PSPClientRepo.ListByUser(userID)` → 每个 `psp_client` 一行 = 每 Server 一行用量；
  `PSPClient.PeriodUsedTotal()` 给本周期用量。跨 Server 求和 = 用户总用量。两个口径都拿得到。
- **为什么不是 per-node**：per-node（per-inbound）需要每节点一个独立 client（即被我们废弃的旧模型），
  且因 Xray 按 email 计量，共享 client 下根本拆不出 per-inbound。**per-Server 是共享模型下最细且最科学的用户用量单位。**
- **历史曲线（P4.4 落地）**：新增 `psp_client_traffic_snapshots`（按 `psp_client` 即 (user,panel) 采样），
  取代旧的 `client_traffic_snapshots`（按 (user,node)）；admin 用户详情页按 Server 分行展示用量 + 历史。
- 与「节点级」是**两条正交轴**：节点级 = 某 Server-inbound 的总流量（跨所有用户）；本节 = 某用户在某 Server 的流量。

## 5. 适配层改动（`internal/adapters/xui` + `internal/ports`）

- `AddClient(ctx, inboundIDs []int, spec)`：`inboundIds` 真正用起来（现在恒 `[]int{inboundID}`，
  [client.go:471](../internal/adapters/xui/client.go#L471)）。
- **新增** `AttachClient(ctx, email, inboundIDs []int)` → `POST /clients/{email}/attach`；
  `DetachClient(ctx, email, inboundIDs []int)` → `.../detach`。
- 入组/退组/加节点改走 **bulk**（`bulkCreate` / `bulkAttach` / `bulkDetach`）——每批**一次 Xray 重启**，
  比现在逐 inbound 写少掉大量重启（用户掉线大减）。
- `GetClient` 已返回 `{client(uuid/password/auth), inboundIds}`（[ports.ClientDetail](../internal/ports/xui.go#L158)）→
  reconcile 直接拿来 1:1 比对。
- `ClientSpec` 已能同时携带 `ID/Password/Auth`（[xui.go:241](../internal/ports/xui.go#L241)），无需改形状。

## 6. sync / reconcile 改动

- **ownership 守卫重 key**：`ensureClientOwned` 从 `(panel, inbound, email)` 改成 `(panel, email)` 或 `(clientID)`。
  「这个 email 在本面板归 PSP 管吗」即可，不再带 inbound。
- **挂载差分**：desired（`psp_client_inbound`）vs actual（`GetClient().inboundIds`）→ attach 缺的、detach 多的。
- **reconcile 变简单**：逐 `psp_client` 与 `/clients/get` 逐字段比（uuid/password/auth/enable/expiry/inboundIds），
  不再有 per-(inbound,email) 的翻译层。
- **sync-task 类型**：`user_resync` / `user_push_config` 语义改为「整理一个 (user,panel) client + 其挂载集」，
  粒度从 per-node 升到 per-(user,panel)。

## 7. Schema 与迁移

- **新表** `psp_client`、`psp_client_inbound`（GORM AutoMigrate）。
- **一次性迁移**（沿用 `psp migrate` 的 sentinel 守卫思路，幂等）：
  1. 按 (userID, panelID[, credClass]) 聚合现有 `user_xui_clients` 行 → 建 `psp_client`（生成/计算 Password=`NewProxyPassword(UUID)`）。
  2. 每条原行的 nodeID → `psp_client_inbound`（flowOverride 从 node.Flow 取）。
  3. 流量计数：N 行求和 → 1 行（§4 迁移）。
  4. **3X-UI 侧重排**：对每个 `psp_client`，`bulkAttach` 到其挂载集 + `bulkDel` 掉多余的旧 per-node email。
     幂等、可重入；失败的进 sync-task 重试。
  5. **imported/legacy client**（运营手建、保留原 email、不走 `ClientEmail` 的，见
     [types.go:237](../internal/domain/types.go#L237)）**不并入**，保持原样，单列。
- **回滚**：迁移前 DB 备份；新旧表并存一个版本窗（旧表只读保留，确认无碍后下一 major 删）。

## 8. 兼容性 / 版本

- `attach/detach/inboundIds` ≥3.1.0 即有、3.3.1 实测在 → **`version.MinXUI` 维持 3.2.0**，无双路。
- compat 矩阵新增一条 entry：`psp_min = v3.9.0`，`min_xui = "3.2.0"`，`max_tested_xui = "3.3.1"`
  （放 [compat/v3.json](compat/v3.json) `entries` 最前）。
- **升级前置**：建议全舰队先到 ≥3.2.0（本就是 floor）；混合舰队按面板逐个迁移（client 级幂等，不要求同刻切）。

## 9. 分阶段实施（都可独立上线，自小到大）

1. **P1 适配层补齐**（纯新增，零破坏）：`AttachClient`/`DetachClient`、`AddClient` 收 `[]int`、接 bulk 端点。
2. **P2 流量改源**（独立 correctness）：节点级读 inbound 计数器；用户级单 client 读。**先解双重计数**。
3. **P3 入组/退组/加节点走 bulk**（拿「少重启」收益，仍不碰 email/ownership 模型）。
4. **P4 模型 + schema + 凭据全对称 + 迁移**（最大块）：`psp_client(_inbound)` 表、email 改 `u{uid}`、
   render 读存值、`DeriveProxyPassword` 退役、一次性迁移。可灰度（先小群组试点）。

## 10. 风险 / 开放问题

- **流量计数断点**：合并那一刻历史 per-node 明细丢失，节点级历史曲线从 inbound 计数器重新起算。可接受，需 UPGRADE 文档明示。
- **SS-2022 128/256 混用**：靠 `credClass` 单列第二 client；需在建号/迁移时检测同 (user,panel) 是否跨 keylen。
- **Trojan / 普通 SS 用户 password 在迁移时变更**：旧模型 Trojan/SS 的 password = UUID；v3.9.0 统一为
  `crypto.NewProxyPassword(UUID)`（= `base64(sha256(UUID))`，所有 password 协议共用）。所以这两类协议的
  **旧订阅需重新拉取一次**（多数客户端定时自更，影响是一次性短暂的）。**不变**的：VLESS/VMess（`id`=UUID）、
  Hysteria2（`auth`=UUID）、SS-2022-256（本就等于统一值，见 `NewProxyPassword` 注释）。迁移前可在 UPGRADE
  note + 邮件提醒里告知受影响用户。
- **节点级流量语义变化**：从「PSP 托管 client 之和」变「inbound 总量」；独占 inbound 无差别，共享 inbound 会含外部 client。
- **imported client**：保持单列、不并入；reconcile 对它仍走旧式 per-row 比对，或标 `unmanaged` 跳过。
- **多面板**：client 是面板内实体，跨面板不共享 → 一个用户 M 个面板 = M 个 `psp_client`，符合预期。

## 11. 切换序列（对抗式验证后的定稿）

> 来源：多智能体「设计 → 对抗攻击」工作流(ground→design→attack)对真实代码的逐行核验。
> **当前状态(已提交)**：模型+repo+planner+provisioner + ResyncMembership 影子 dual-write 全部就绪;
> **但生产无任何读者**——render 仍 `DeriveProxyPassword`、sync 仍 per-node、traffic 仍读 ownership。
> 下列「洞」**目前都不是 active bug**(没人读 `psp_client`、3X-UI 里还没共享 client),全是**切换期**地雷。

**承重不变式(唯一总护栏)**:render 只有在「该 (email,uuid,password) 已确认 attach 到该 node 的 inbound 上(3X-UI 实读回确认)」时才对该 node 发共享凭据;否则回退到**与今天逐字节一致**的 legacy 派生路径。最坏情况永远是「保持/回退 legacy + 旧 per-node client 仍在 = 永不断网」。

**严格阶段顺序(不可重排)**:`create-in-3xui → flip-render → flip-traffic → remove-old`。门控用**每-(client,node) provisioned 谓词**(非全局 flag,非"行存在")+ 一组 `UISettings` kill-switch(`SubRenderUseSharedClient` 等)做即时回退(≤60s,render 缓存 TTL)。**一次性大爆炸方案已否决**(无共存窗口 → Trojan/SS 密码变更对所有人同时硬断)。

- **Stage 0 — 全量回填**:dual-write 只在 resync 触发,存量未 resync 用户无行。一次性 idempotent 跑 `clientprov.SyncUser` 覆盖所有用户(纯 PSP DB 写,零 3X-UI)。SS-2022 节点须先解析 live method(见洞#6)。
- **Stage 1 — 在 3X-UI 建共享 client + 全生命周期(关键)**:新建 reconcile 服务,对每个 psp_client `AddClientToInbounds`(email=`u{uid}`、id=UUID、password=stored);**并把 enable/expire/quota-floor/uuid 轮换全部接到共享 client**(见洞#1)。仅在 `GetClient` 读回确认 attach 后才标 provisioned(**每-attachment 粒度**,见洞#2)。旧 per-node client 不动 → 两套并存、无人引用新的 → 不断网、不双计。
- **Stage 2 — render 切共享(每-node + 回退)**:`clientFor(node)` 仅在该 node 有 **provisioned** attachment 时返回 stored 凭据;否则 legacy。三个 dispatcher(`emitProxy`/`emitSingBoxOutbound`/`buildURI`)改收显式 `(uuid,password)`。kill-switch 全局兜底。
- **Stage 3 — traffic 切共享(须紧跟 render,且原子对齐)**:按 email **一次**读共享 client 聚合,折进 `psp_client` 计数(`UpdateCounters`),**绝不**给共享 email 建 N 条 ownership 行(否则 3X-UI 每 inbound 回显同一聚合 → N 倍超计)。Stage2→3 间隙是欠计窗口,须**按用户对齐**两者翻转。基线:共享 client 是全新的(计数从 0 起),bootstrap=current 正确。
- **Stage 4 — 删旧(唯一不可逆,放最后 + soak)**:soak ≥1 个 `SubUpdateIntervalHours`(默认 24h)让 Trojan/SS 旧订阅都已刷到新密码后,再 `DelClientByEmail` 旧 per-node + 删 ownership 行。删前把旧计数并进 psp_client lifetime。

### 对抗审查发现的 7 个洞(实现切换前必须处理)

1. **(critical) 生命周期/enforcement 全在 ownership**:`SetEnabledAndSync`/`ResetUUIDAndSync`/expiry/quota 都只推 per-node client。render 切共享后被禁用/到期/超量的用户**仍能用**(面板只关了没人用的旧 client)。→ **Stage 1 必须把共享 client 的 enable/expire/quota/uuid 全接上,且早于任何 render flip**。
2. **per-client bool 不够**:跨面板 partial attach → 须**每-(client,node)** 确认 attach(在 `psp_client_inbounds` 上记 provisioned,或 render 时与 `GetClient().InboundIDs` 取交集)。
3. **plain-SS 可能单密码**:非-2022 SS inbound 若服务端单密码,"两 client 共存"兜底**可能不成立** → plain-SS 切换前须逐 inbound 确认支持多密码,否则强制刷新或暂不并入。VLESS/VMess/Hy2/SS-2022 均不变、安全。
4. **双计 + 基线**:共享 client 跨 N inbound 回显同一聚合 → 必须按 email 读一次;且 render/traffic 翻转须原子对齐(否则欠计窗口)。
5. **EmailDomain 变更孤儿**:`Upsert` 以 `(panel,email)` 为键;迁移期改 `EmailDomain` 会建重复行 + 3X-UI 孤儿。→ 迁移期**冻结 EmailDomain**,或 `Upsert` 改以 `(panel,user,credClass)` 为键、email 当可变列。
6. **未捕获 SS-2022-128 误分类**:空 InboundSettings → method 未知 → 误入 credClass 0(32B) → Xray 拒 16B inbound。→ 回填/建号对 SS-2022 节点**硬失败**而非跳过,或先解析 live settings。
7. **dual-write 写写竞争**:dual-write 每次 resync 都 delete/replace `psp_client` 行;一旦有读者(reconcile),会与之竞争 + 冲掉 provisioned 标志。→ Stage 1 起 **reconcile 成为 provisioning 状态唯一写者**,dual-write 改为「入队 reconcile job」而非直接改。
