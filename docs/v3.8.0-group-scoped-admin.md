# v3.8.0 分组化管理（Group-Scoped Administration）设计稿

> 状态：设计稿（先文档、后实现）。
> 约定：docs 中文；代码注释英文；标识符沿用代码原名（`UISettings`/`Group`…）。
> 单一真相源 + drift 防护；严格 TDD（red→green）；自用项目无迁移脚手架（改 struct 走 AutoMigrate，新增表/列直接加）。
> 跨方言可移植：不用 `type:json`（Postgres 拒空串），upsert 走 GORM `clause.OnConflict`。

把「全局一份」的设置升级为 **全局默认 + 各 Group 覆盖**，并把管理控制台对齐 Google Workspace Admin（scope 选择器 + 继承/覆盖）。本稿是该改造的技术内核与分期。

---

## 1. 三项已定决策（固定，后续不回退）

1. **作用域模型 = 两级**：全局默认（“All users”）+ 扁平 per-`Group` 覆盖。**不做 OU 树**（无层级继承/优先级）。
2. **成员模型 = 一人一组**：`domain.User.GroupID int64`（单一外键）。无多组成员、无优先级决冲。
3. **UI = 全盘 Workspace 式控制台重构**：左侧 scope 选择器（All users 默认 / 各 Group），右侧该 scope 设定，逐字段 **继承 / 覆盖 / 恢复默认**。

附带：**Email 作为可满足强制2FA 的因子**（见 §7）——按用户确认纳入，且天然是一条 per-group 策略。

### 1b. 本轮补充已决（2026-06）

4. **Dashboard 保留为管理首页**：现有主页 Dashboard 不动、置顶为独立导航项，不并进 Reporting（§6.1 据此调整）。
5. **注册作用域 = 全局 + 落地组设置**：注册策略（开关/投递/域名/默认配额）留全局；「新用户进哪个组」沿用**已存在的全局设置 `RegistrationDefaultGroupID`**（与 SSO 的 `default_group_slug` 同思路：一个全局「默认落地组」）。§10-2 据此收口——注册**不**做 per-group 覆盖。
6. **版本 = v3.8.0（不升 v4）**：本稿纯 additive（`scope_settings` 走 AutoMigrate 新表，无迁移函数、无数据搬运）→ 按「无 DB 迁移即非重构」**不升 v4**；又因是独立的「重要新功能」（按 semver minor）单列 **v3.8.0**，让 v3.7.0「账号安全」先发稳定版、主题不被拖。`v4-inbound-ownership`（动 reconcile/render 热路径、属重构）仍单列 v4。
7. **Email-as-factor 用新字段** `TwoFAEmailAsFactor`（§7）——确认采用，不复用 `TwoFAAllowEmail`。
8. **删 `User.Require2FA`、作用域锁死两级**：分组化后 2FA 强制由 `Require2FAForStaff`（全局/staff）+ `Group.Require2FA`（组）覆盖，per-user 那一列冗余且与「两级」相悖 → 移除（DB 列 AutoMigrate 不自动删、留孤儿列无害，同 v3.7.0-beta.16 孤儿 KV 处理）。`MustEnroll` 收敛为 `Require2FAForStaff ∨ Group.Require2FA`。**区分**：删的只是*策略层* per-user；per-user *数据*（配额/到期/凭证）一直在 `User` 行、与设置系统无关、不受影响。§6.4 随之退役。

**DB 审查结论（多 agent + 对抗验证，2026-06）**：现有 settings 存法与本稿 `scope_settings` 叠加层**均为此规模下最优、不改 schema**（逐一否决了「每字段一列」「每组一个 JSON blob」「每组一宽行」三种替代）。两条改进已并入下文——§5 缓存改**单-gen seqlock**（消除撕裂读）、Phase 0 置顶补 `settings_cache_test.go`（TDD）；§10-1 / §10-3 据此收口。

零迁移：现有全局 `settings` 行即「All users」默认；`scope_settings` 空表 = 全部继承。

## 2. 现状锚点（grounding）

- 全部设置是一个宽 struct `ports.UISettings`（**84 字段**），扁平进 KV 表 `settings`（`settingRow{ID, Type size:32, Name size:128, Value text, Encrypted bool, UpdatedAt}`，唯一索引 `uk_setting_kv(type,name)`），映射真相源是 `settingDescriptors(*UISettings)`（每字段一个 `settingDescriptor`，闭包持有结构体字段指针）。仓库 `kvSettingsRepo`（`internal/adapters/sqlstore/settings_kv_repo.go`），接口 `ports.SettingsRepo`：`Load(ctx, defaults)` / `Save(ctx, s)`。
- 缓存当前是**全局单份**：`cachingSettingsRepo`（`settings_cache.go`）持有一个 `cached *UISettings` + `gen` 失效计数；`Save` 置 `cached=nil; gen++`（TTL=0，改完即时可见，`/sub/:token` 热路径依赖此）。
- **唯一已存在的 per-scope 级联**是 `require_2fa`：`authpolicy.Service.MustEnroll`（`authpolicy.go:40`）今天 = `User.Require2FA` ∨（`Require2FAForStaff`，仅 admin/operator）∨ `Group.Require2FA`。**这是本稿要泛化的样板**；本稿同时**删掉 `User.Require2FA`**（决策 8），级联收敛为 `Require2FAForStaff ∨ Group.Require2FA`（staff全局 ∨ 组），干净两级。
- `Group` 已有的 per-group 行为：`TagFilter`（选节点，`group.Service.NodesFor`）、`Layout`（渲染排序/分隔）、`Require2FA`。**无** per-group 配额/到期/角色/规则集。
- 一人一组：`User.GroupID`；`GroupID==0` = 无组（authpolicy 容忍、render 会报错，实践中活跃用户都有组）。

## 3. 全局 vs 分组 划分

判定规则：某字段**可分组**当且仅当——有消费方用它做 **per-user/per-group 策略决定**，且其值按受众不同不会破坏进程级不变量。凡进程 ticker / 连接·密钥 / 面板级身份 / 限流·留存窗口，一律 **全局**。

### 3a. 可分组策略（每 Group 可覆盖）

| 字段 / 组 | 消费方与理由 |
|---|---|
| `DisallowUserLocalLogin` / `DisallowUserPasswordChange` / `AllowUserPersonalRules` | `auth_local`/`user_me` 按账号判；「本组必须走 SSO / 禁改密 / 允许自定义规则」是自然的分组策略 |
| `TOTPEnabled` / `PasskeyEnabled` / `PasskeyPasswordless` / `TwoFAAllowEmail` / `TwoFAEmailResendCooldownSec` / `Require2FAForStaff` | 2FA 启用/方式/强制——`twofa`/`passkey`/`login2fa`/`authpolicy` 全是 per-user 决策；`Require2FAForStaff` 已在 `MustEnroll` 级联里 |
| `LockoutEnabled` / `LockoutThreshold` / `LockoutWindowMinutes` / `LockoutDurationMinutes` / `LockoutScope` | `loginguard.Evaluate` 对单 `(ip,upn)` 判。**注意**：登录在识别身份前发生，见 §10-1 |
| `CaptchaEnabled` / `CaptchaTrigger` / `CaptchaFailThreshold` / `CaptchaRegisterEnabled` / `CaptchaForgotEnabled` | 表单验证码**启用/触发**策略（per-context）。**`CaptchaProvider`/`SiteKey`/`SecretKey` 留全局**（实现+密钥=连接配置） |
| `PasswordRecoveryEnabled` / `PasswordRecoveryDelivery` | `recovery.RequestReset` per-user |
| 紧急访问：`EmergencyAccessEnabled` / `Hours` / `MaxCount` / `QuotaGB` | `user.UseEmergencyAccess`/`emergencyFloor`/`traffic` 全 per-user |
| 订阅策略：`SubBlockAutoDisable` / `SubBlockAutoDisableCount` / `SubBlockNotifyUser` / `SubBlockNotifyMaxPerDay` / `SubRegionFlagPrefix` / `SubUpdateIntervalHours` / `SubProfileNameTemplate` | `sub.Get`/`render.RenderForUser` per-user 封禁与呈现 |
| 通知阈值：`ExpireBeforeDays` / `TrafficRemainPercent` | `mailer.processUser` per-user 提醒阈值——最干净的分组场景 |
| `LoginMode` | **待决，暂全局**：登录页在识别身份前就要据它决定 SSO 跳转；若要「本组 SSO-only」与 `DisallowUserLocalLogin` 重叠。见 §10-1 |

### 3b. 仅全局（无覆盖）

| 字段(组) | 原因 |
|---|---|
| `JWTAccessTTLMinutes`/`JWTRefreshTTLMinutes`/`JWTIssuer` | 令牌签发身份，进程级（`app.go` 闭包/`jwtutil`） |
| `CronTrafficPullMinutes`/`CronReconcileMinutes`/`MaxPanelConcurrency` | 进程 ticker / 扇出上限 |
| `SubPerIPPerMin`/`LoginPerIPPerMin` | 启动期播种的 per-IP 限流（先于身份） |
| `SubPath` | 单一订阅路径前缀 |
| `*RetentionDays`（audit/synctask/traffic-history/mail/sub-log/auth-event） | DB 留存窗口，面板级 prune 循环 |
| `Timezone` | 系统日界，单进程 |
| `EmailDomain` | 3X-UI `client.email` 后缀，全 fleet 唯一/稳定 |
| `SubBaseURL` | 面板唯一公网址；又是 WebAuthn RP-ID 来源 + 找回/注册链接信任基。必须全局（RP-ID、防链接投毒） |
| 品牌：`SiteTitle`/`AppTitle`(`BrandName()`)/`IconURL`/`LogoURL`/`LogoURLDark`/`FooterText`/`ThemeColor` | 面板身份（TOTP issuer、RP 显示名、邮件头、登录页）。**待决**：per-group 门户 logo/主题是将来可加的装饰覆盖（§10-6），但 RP-ID/issuer 必须全局 |
| GeoIP 全 8 字段 / `CertRenewBeforeDays`·`CertRenewCheckIntervalHours` | 单一离线库+下载凭据 / 单证书 fleet 续期节律 |
| `SubClients`/`SubClientFilterMode`/legacy / `QuickLinks` / `GlobalAnnouncement` | 客户端注册表 + 门户级公告/快捷链接（升级为独立页，§6.1）。per-group 公告将来可加（§10-6） |
| `CaptchaProvider`/`CaptchaSiteKey`/`CaptchaSecretKey`(enc) / `GeoIPUpdateToken`(enc) | 实现/密钥=连接配置；两个加密字段均全局 |
| 注册：`RegistrationEnabled` / `RegistrationAllowUnverified` / `RegistrationDelivery` / `RegistrationEmailDomains` / `RegistrationDefaultTrafficGB` / `RegistrationDefaultExpireDays` / `RegistrationDefaultGroupID` | **决策 5**：注册是面板级开放策略，注册时用户尚无组 → 全部全局。`RegistrationDefaultGroupID` 即「新用户落地组」（对齐 SSO `default_group_slug`）；落地后该组的 per-group 覆盖（§3a）自然生效 |

> 加密字段（`captcha_secret_key`、`geo_ip_update_token`）都在 3b → 实践中**永不写加密覆盖**；**已决（§10-3）：`SetOverride` 在边界直接拒绝 `Encrypted=true` 的覆盖**。

## 4. 数据模型：`scope_settings`（叠加在 `settings` 之上）

新增一张与 `settingRow` 同形、外加 scope key 的稀疏 KV 表。**全局 `settings` 不动**（仍是「All users」的真相）。

```go
// internal/adapters/sqlstore/scope_settings_repo.go
type scopeSettingRow struct {
    ID        int64  `gorm:"primaryKey;autoIncrement"`
    ScopeType string `gorm:"size:16;not null;uniqueIndex:uk_scope_setting,priority:1"` // "group"（预留 "user"）
    ScopeID   int64  `gorm:"not null;uniqueIndex:uk_scope_setting,priority:2"`         // groups_.id
    Type      string `gorm:"size:32;not null;uniqueIndex:uk_scope_setting,priority:3"`
    Name      string `gorm:"size:128;not null;uniqueIndex:uk_scope_setting,priority:4"`
    Value     string `gorm:"type:text"`
    Encrypted bool   `gorm:"not null;default:false"`
    UpdatedAt time.Time
}
func (scopeSettingRow) TableName() string { return "scope_settings" }
```

要点（每条都绑现有约束）：
- **与 `settingRow` 同形** → 复用现有 `settingDescriptors`、字段 helper（`strField`/`intField`/`boolField`/`jsonField`/`encStrField`）、`encryptSecret`/`decryptSecret`、`KnownSettingNames()`，只换行来源。
- **稀疏**：某 `(scope_type, scope_id, key)` 只有被显式覆盖才有行；缺行 = 继承。这正是零迁移的原因（空表即全继承）。
- **复合唯一** `uk_scope_setting(ScopeType,ScopeID,Type,Name)`，跨方言；upsert 用 `clause.OnConflict`。
- **`ScopeType` 用字符串非枚举**：本稿**只写 `"group"`**（作用域锁两级、无 per-user 策略层，决策 8）；字符串列只是不花成本地留扩展位，**不承诺**将来做 user 层。
- `schema.go` AutoMigrate 收，无迁移文件。

`ports.ScopeSettingsRepo`（薄覆盖层；`ports.SettingsRepo` **不变**，老调用者继续拿全局值）：
```go
type ScopeOverride struct { Type, Name, Value string; Encrypted bool } // Name 必须 ∈ KnownSettingNames()
type ScopeSettingsRepo interface {
    ListOverrides(ctx, scopeType string, scopeID int64) ([]ScopeOverride, error) // 稀疏
    SetOverride(ctx, scopeType string, scopeID int64, o ScopeOverride) error // 拒 Encrypted=true（§10-3）
    DeleteOverride(ctx, scopeType string, scopeID int64, typ, name string) error // = 恢复继承
    DeleteScope(ctx, scopeType string, scopeID int64) error                       // 删组时清覆盖
}
```

## 5. 生效解析

一人一组 → **两行合并**，无优先级树：
```
effective(group) = applyOverrides( globalDefault, overridesOf(groupID) )
effective(user)  = effective(user.GroupID)        // GroupID==0 ⇒ 纯全局
```
`applyOverrides` 用与 `kvSettingsRepo.Load` **同一套** `settingDescriptor.Unmarshal`，把每条 `ScopeOverride` 写进全局 `UISettings` 的副本；缺的 key 保留全局值（稀疏=继承）。覆盖必须作用在**已 `applyUISettingsDefaults` 过的全局基**之上（避免组把已 floor 的字段重新归零；不再 per-scope 跑 key-presence 留存逻辑）。

新增 `ScopedSettings` 服务（包住 `SettingsRepo` + `ScopeSettingsRepo`）：
```go
type ScopedSettings interface {
    Load(ctx, defaults) (UISettings, error)                          // 全局，等同今天
    LoadForGroup(ctx, groupID int64, defaults) (UISettings, error)   // 全局 ⊕ 组覆盖
    LoadForUser(ctx, u *domain.User, defaults) (UISettings, error)   // = LoadForGroup(u.GroupID)
}
```

**消费方迁移（三档，按爆炸半径从小到大）**：
1. **全局消费方（3b）不动**：`app.go` 循环、`router.go` 限流、geo/cert/留存 prune、`paneltz`、`EmailDomain` 用户、JWT 闭包——继续 `Load`。
2. **已持有 user 的 per-user 消费方**：`Load → LoadForUser`（机械替换）。`authpolicy.MustEnroll`（本就同时加载 settings + group，收敛进 `LoadForUser`：删掉它手写的组 OR **及** `User.Require2FA` 项——决策 8）、`user_me`、`user.UseEmergencyAccess`/`emergencyFloor`、`traffic`、`sub.Get`、`render.RenderForUser`、`mailer.processUser`、`recovery`、`twofa`/`passkey`/`login2fa`。
3. **识别身份前的登录消费方**（`loginguard.Evaluate`、captcha gate）：见 §10-1，**v3.8.0 保持全局**，后续再 promote。

`authpolicy.Deps` 是模板：已含 `GroupGetter`+`SettingsLoader`，v3.8.0 把 `SettingsLoader.Load` 换成合并版 `LoadForUser` 即可。

**缓存（per-scope，单-gen seqlock）**：**不引入第二个 generation**（双 gen 会在「全局 `Save` + 某组覆盖写」并发时把*新全局基*与*旧覆盖集*拼一起 = 撕裂读；因可覆盖字段含 2FA 强制策略，撕裂读会算出*错误的生效安全设置*——DB 审查标定的唯一真设计缺口）。改为：**全局 `Save` 和任何 scope 覆盖写都 bump 同一个 `gen`**，per-scope 生效值退化为现有单-gen 纪律的复用——seqlock 式乐观读：

```
for {
    g0 := gen.Load()
    base := globalCache.Load(ctx)     // 现有全局 cache，本身已是一致快照
    ovr  := overridesFor(scopeID)     // 该组稀疏覆盖集（0–10 条）
    eff  := applyOverrides(base, ovr) // 复用 settingDescriptor.Unmarshal
    if gen.Load() == g0 {             // 读窗口内无写 → base 与 ovr 必同一时刻
        scopeCache.put(scopeID, g0, eff); return eff
    }                                 // 否则重算（tens-of-users 规模几乎不触发）
}
```

返回值整段在「`gen` 未变」窗口内算出 → 基与覆盖必然同一时刻、**结构上不可能撕裂**。写侧 `Save`/`SetOverride`/`DeleteOverride` 均 `gen++`；代价是改某组覆盖会让别组 cache 一并失效一次，此规模无所谓。`DeleteScope` 接进 `group.Service.Delete`（已拒非空组）。**前置**：Phase 0 先给现有单-gen 纪律补 `settings_cache_test.go`（今天零覆盖），再据此复用。

### 6.2 scope 选择器（可分组类目的左轨）
```
┌──────────────┬───────────────────────────────────────────┐
```

- `require_2fa` 是已能工作的样板（删 `User.Require2FA` 后为 staff∨组两级），v3.8.0 把它泛化成统一覆盖面。

### 6.3 Group 详情成为 scope 枢纽
GroupsView 今天只是节点过滤表（唯一策略 `require_2fa`）。v3.8.0 组详情加 tab：**Members** / **Node access**（现 `tag_filter`+`Layout`）/ **Policies**（该组的覆盖面，与 Settings scope 轨同组件）。组级 `require_2fa` 移进 Policies、退役 GroupsView 那个独立开关；账号安全抽屉里那个 per-user `require_2fa` 开关一并移除（决策 8）。

### 6.4 无 per-user 策略层（决策 8）
作用域**锁死两级**（全局 ⊕ 组）。原 `AccountSecurityDrawer` 的 per-user `require_2fa`（`User.Require2FA`）**删除**——分组后由 `Require2FAForStaff` + `Group.Require2FA` 覆盖，per-user 那层冗余。`scope_settings.ScopeType` 仍是字符串（留扩展位、不花成本），但**不写 `"user"`、不承诺三级合并**。per-user 的*数据*（配额/到期/凭证）继续在 `User` 行上，与设置系统无关。

## 7. 关联改造：Email 作为可满足强制2FA 的因子（Q2=能）

现状：`authpolicy.MustEnroll` 仅 TOTP 或 passkey 算「已满足」；Email 只是已绑因子后的登录备选（`auth_local` 的 2FA 闸是 `u.TOTPEnabled || hasPasskey`）。用户要：**强制2FA 时，开了 Email 即可用 Email 满足**（对没有验证器 App 的用户友好）。

改动（天然 per-group，挂在 §3a 的 2FA 策略下）：
- **新增一档策略** `TwoFAEmailAsFactor`（bool，可分组）：开启后
  - `MustEnroll` 的「已满足」加一支：`TwoFAEmailAsFactor && u.Email != ""`（有邮箱即视为已具备一个因子）。
  - 登录 2FA 闸（`auth_local`）触发条件加 `|| (effective.TwoFAEmailAsFactor && u.Email != "")` → 这类用户登录走 2FA 挑战、收 Email 验证码。
  - `availableTwoFAMethods` 在无 TOTP/passkey 时也提供 email。
- **安全声明**（写进 UI 文案 + 文档）：Email-only 2FA 较弱（掌握密码+邮箱即可过）；默认关；建议仅对低敏分组开。
- 与现有 `TwoFAAllowEmail` 区分：`TwoFAAllowEmail` = 「已有真因子的人**额外**可用 Email 备选」；`TwoFAEmailAsFactor` = 「Email **本身**可满足要求」。后者蕴含前者。
- TDD：MustEnroll 在「仅 Email-as-factor + 有邮箱」下返回已满足；无邮箱仍需强制；关闭时回到现状。

> 备选实现：不新增字段，直接让 `TwoFAAllowEmail` 兼任「可满足」——更省字段但语义混（“备选”≠“可满足”），且无法只开备选不开 satisfies。**已决（决策 7）：新增 `TwoFAEmailAsFactor`**，语义清晰、可分组。

## 10. 风险与待决问题

1. ~~识别身份前策略~~ **已决（DB 审查确认）**：`lockout / captcha / LoginMode` 在用户（组）解析前跑，**v3.8.0 保持全局**、不迁这些消费方。将来要提 per-group 必须 UPN→user→group 预取且对**不存在/攻击者构造的 UPN** fail-safe 回全局（否则既错又成枚举面），并先 benchmark 每登录多一次查询。`LockoutScope=ip` 根本无 user → 永远全局。
2. ~~注册作用域是前向引用~~ **已决（决策 5）**：注册整组留全局，不做 per-group 覆盖；「新用户落地组」由全局 `RegistrationDefaultGroupID` 指定（对齐 SSO `default_group_slug`）。彻底避开「组覆盖 `RegistrationEnabled` 的鸡生蛋」。落地后该组的 per-group 策略（§3a）自然对该用户生效。
3. ~~加密覆盖虽支持但不用~~ **已决（DB 审查）**：`ScopeSettingsRepo.SetOverride` 在边界**直接拒绝** `Encrypted=true` 的覆盖（两个加密字段都全局、永无合法加密覆盖）——关掉一条永不被正确走到的分支，避免将来某字段误标 encrypted+overridable 时静默走未测路径。
4. **`applyUISettingsDefaults` 次序**：覆盖必须作用在已默认化的全局基上，且不 per-scope 重跑 key-presence（730/7/90）。解析器在全局 `applyUISettingsDefaults` **之后**应用覆盖。
5. ~~缓存双 generation 风险~~ **已解（DB 审查的唯一真缺口）**：不引入第二个 generation——全局 `Save` 与任何 scope 覆盖写共用**单 `gen`** + seqlock 读（§5），结构上消除「新全局基 ⊕ 旧覆盖集」撕裂读。配 `settings_cache_test.go`（先钉单-gen 纪律）+ 交错 `-race` 测试。
6. **品牌/门户 per-group（暂缓但预期）**：`LogoURL`/`ThemeColor`/`GlobalAnnouncement`/`QuickLinks` v3.8.0 仍全局，但是明显的 per-group 装饰诉求；`SubBaseURL`/`BrandName()` 必须全局（RP-ID/issuer）。`scope_settings` 任意 key 都支持，将来落装饰无需改模型。
7. **生效来源标识 = 两级**：徽标只有「继承自 All users / 此处覆盖」两态。删 `User.Require2FA`（决策 8）后**再无 per-user 策略层**，不存在第三层合并问题。
8. **`GroupID==0`**：解析器定义 0 → 纯全局（不查覆盖），对齐 authpolicy 现有 fail-safe。
